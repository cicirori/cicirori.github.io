概览

最直觉的 I/O 模型是, 一个进程在同一时刻对一个文件描述符进行 I/O 操作, 每个 I/O 系统调用阻塞直至数据传输完成. 但是这种简单的阻塞式的 I/O 模型在很多场景下并不适用.

  • 尽可能不阻塞地检查一个文件描述符是能进行 I/O.
  • 监视多个文件描述符.

于是乎, 就有了本文要讨论的这三种模型 :

  • I/O 多路复用 (select() 以及 poll() 系统调用)
  • 信号驱动 I/O
  • Linux 特有的 epoll() API

这三种方式实际上都是为了实现同一个目标, 同时检查多个文件描述符, 查看他们是否准备好了执行 I/O 操作. 这里的准备好, 实际上指的是在这个文件描述符上执行的 I/O 是非阻塞的(包括返回错误), 因为模型只是对文件描述符检查, 而不进行数据传输.

I/O 多路复用

这里指的是来自 BSD 系统中的 select(), 和 System V 中的 poll(), 这两个系统调用的功能几乎相同. 这两个系统调用都是阻塞式的, 要么满足终止条件, 要么产生超时.

select() 系统调用

select() 原型如下 :

#include <sys/time.h>
#include <sys/select.h>

int select(int nfds, fd_set *readfds, fd_set *writefds, fd_set * exceptfds, struct timeval *timeout);

nfds 是select 监控的文件描述符数量, readfds, writefds, exceptfds 是存放感兴趣的文件描述符的集合, 以及返回结果的数据结构. timeout 超时阈值, 可设置为 NULL 一直阻塞.

使用 select() 检查一些文件描述符的操作如下:

  1. 初始化 readfds, writefds 和 exceptfds, 把对应的文件描述符加入对应的集合, 设置合适的 nfds 和 timeout 值.
  2. 执行 select().
  3. 阻塞至 select 执行完成.
  4. 检查结果.

poll() 系统调用

poll 和 select 的主要区别是存放文件描述符用的数据结构, 下面是 poll 的原型:

#include <poll.h>

int poll(struct pollfd fds[], nfds_t nfds, int timeout);

可以看出和 select 的参数相似, struct pollfd 的定义为:

struct pollfd {
	int fd;					/* File descriptor */
	short events;			/* Requested events bit mask */
	short revents;			/* Returned events bit mask */
};

这里的 events 代表了 fd 感兴趣的事件, revents 代表了返回的结果.

相比于 select 只支持 write, read 和 exception, poll 支持的事件种类比较多:

POLLIN • • 可读的非高优先级数据
POLLRDNORM • • 等同于POLLIN
POLLRDBAND • • 可读取的优先数据(Linux 未使用)
POLLPRI • • 可读的高优先级数据
POLLRDHUP • • 对端套接字关闭
POLLOUT • • 可写普通数据

POLLWRBAND • • 可写的优先数据

普通文件

指向普通文件的 FD 总会返回 readable / POLLIN 和 writeable / POLLOUT. 因为如前文提到的就绪实际上指的是非阻塞, 对普通文件的 write 和 read 总是可用的或者直接返回异常或错 误.

select 和 poll 异同点

实现细节

两种调用都使用了一系列相同的 Linux 内核的 poll 例程. 这里的 poll 指的不是 poll() 系统调用. 这里每个 poll 例程返回一个文件描述符的信息, 信息以和系统调用 poll() 相同的 revent 掩码来表示, select 的三种结果来自 revent 的宏定义:

#define POLLIN_SET		(POLLRDNORM | POLLRDBAND | POLLIN | POLLHUP | POLLERR)
						/* Ready for reading */
#define POLLOUT_SET		(POLLWRBAND | POLLWRNORM | POLLOUT | POLLERR)
						/* Ready for writing */
#define POLLEX_SET 		(POLLPRI)
						/* Exceptional condition */

API 差异

  1. select 使用的 fd_set 数据结构有一个上界, 默认为 1024, 需要通过重新编译 Linux 来改变. poll 则没有明确的限制.
  2. 因为 select 的 fd_set 不仅是传入的参数, 还是结果, 所以如果要在循环中使用 select, 每次都需要重新初始化. poll 则不需要.
  3. select 提供的 timeout 精度是微秒级, 比 poll 提供的毫秒级更为精确.
  4. 如果监控的文件描述符关闭了, poll 会在revent 中告知我们是那一个, 而 select 只是返回 -1 并把 errno 设置为 EBADF.

可移植性

历史上, select 应用更为广泛, 现如今都存在于 SUSv3 标准中, 且广泛存在于现代的 Unix 实现. poll 的实现更为多样一些.

性能

当下面两条任意一条满足时, poll 和 select 性能相近:

  1. 待检查的文件描述符范围较小 (最大文件描述符数值较小)
  2. 大量文件描述符等待检查, 但是分布较为密集.

当被检查的文件描述符较为稀疏的话, poll 的性能会有明显优势. 因为, select 总要检查 0 - nfds-1 之间的文件描述符, 而 poll 只检查存在的 fd.

存在的问题

  1. 每次进行这两个调用都会对需要检测的文件描述符进行检测.
  2. 需要进行用户空间和内核空间的数据结构交换.
  3. 每次程序都要依次检查返回的结果.

所以随着需要检查的文件描述符数量增加, 这些问题会造成不可避免的性能问题.

信号驱动 I/O

一个很自然的想法就是我们不去一个一个检查, 而是一旦 I/O就绪, 我们就通知程序. 信号驱动 I/O 就是这样一个模型. 这里的信号就是 Linux Kernel 的 signal. 一个信号驱动 I/O 简单流程如下:

  1. 配置信号处理函数, 默认的通知信号是 SIGIO.

  2. 设置文件描述符的 owner, owner 会在 I/O 就绪的时候收到收到信号, 通常设置为调用进程.

  3. 通过设置 O_NONBLOCK 启用非阻塞 I/O.

  4. 打开 O_ASYNC 启动信号驱动 I/O. 和前一步可以合并:

    flags = fcntl(fd, F_GETFL);                  /* Get current flags */
fcntl(fd, F_SETFL, flags | O_ASYNC | O_NONBLOCK);

  5. 调用进程现在可以执行自己的任务, 一旦 I/O 就绪, 内核会向之前设置的 owner 发送信号, 调用第一步中设置的handler.

  6. 信号驱动 I/O 提供了边缘触发通知, 这意味着一旦进程被通知就最好尽可能的进行多的 I/O. 如果 I/O 是非阻塞的话, 尽可能多就意味着进行到出现 EAGAIN 或者 EWOULDBLOCK 为止.

信号驱动 I/O 的优化

在需要同时监控大量文件描述符( 几千个) 的时候, 信号方式对比 select 和 poll 会有很大的性能优势. 因为 Linux kernel “记住” 了需要监控的 文件描述符的一个列表, 并且仅当 I/O 操作发生的时候通知进程. 所以 信号驱动 I/O 的性能与 I/O 事件的数量息息相关, 而不是文件描述符的数量.

为了发挥信号方式的优点, 我们使用专属于 Linux 的 fcntl() F_SETSIG 来指定一个实时信号, 当 I/O 就绪时, 这个信号取代 SIGIO 被发送. 因为 SIGIO 是标准的非排队信号, 一旦多个信号同时发出, 发生阻塞, 后续的 SIGIO 信号会被丢弃. 而我们的实时信号可以挂上队列, 进行处理. 需要注意, 队列长度是有限的, 有发生溢出的可能.

epoll

epoll 是一个 Linux 特定的 API, 相比其他方式, 它有两个主要优点:

  1. 在文件描述符数量多的情况下拥有比 poll, epoll 更好的性能.
  2. 同时支持水平触发通知和边缘触发通知.

epoll 的性能和信号驱动 I/O 接近, 但是相比信号方式, epoll 有如下优点:

  1. 避免了复杂的信号处理.
  2. 在指定我们想要监控的 I/O 类型上有更好的灵活性.

为了解决 poll, select 遇到的问题, epoll 在内核空间维护两种数据结构:

  1. 记录进程的文件描述符感兴趣的 I/O 操作的 interest list
  2. 记录 I/O 就绪的文件描述符的 ready list.

API 调用

  1. epoll_create() 建立一个epoll 实例, 返回指向这个实例的文件描述符.
  2. epoll_ctl() 维护对应实例, 相关的兴趣列表.
  3. epoll_wait() 返回实例的read list.

具体的声明不再赘述.

实现

数据结构与布局

实现的核心是存放在 Linux 内核的数据结构, struct eventpoll:

struct eventpoll{
    ....
    struct rb_root  rbr;				// 双向链表 存放readylist的头结点
    struct list_head rdlist;			// 红黑树 存放interestlist的根
    ....
};

每个事件都会由 struct epitem 表示:

struct epitem{
    struct rb_node  rbn;//红黑树节点
    struct list_head    rdllink;//双向链表节点
    struct epoll_filefd  ffd;  //事件句柄信息
    struct eventpoll *ep;    //指向其所属的eventpoll对象
    struct epoll_event event; //期待发生的事件类型

最后形成的格局就是, interestlist 是个红黑树, readylist 是个双向链表, readylist 是 interestlist 的子集, 这两个结构的 root/head 都在epoll 实例对应的 eventpoll 中, 由一个文件描述符指向这个eventpoll.

最后形成的效果就是, 这两幅图的结合 (手动滑稽).:

第一张图很好的展示了 readylist 和 interestlist 的数据结构, 实际上readylist 和 interestlist 在空间上又是重合的.

回调 事件驱动

光有上节的数据结构也不足以让 epoll 性能如此优秀, 如果 kernel 还是不断的 poll 文件描述符, 那么也无非比 poll调用, select调用省了个传递数据结构的时间.

每当 epoll_ctl() 传进来一个新的事件的时候, kernel 会在相应的事件中断注册回调函数, 一旦事件发生, 就把interestlist 中对应的事件挂到readylsit之后. 不必等到epoll_wait 的时候再去挨个poll.

具体的细节待进一步研究.

epoll 语义的进一步探究

我们之前提到 epoll_create返回的是文件描述符, 但是与 eventpoll 对应的是系统范围的文件描述而不是文件描述符

文件描述符, 文件描述和i-node 关系如下:

这就造成了一些微妙的差异:

  1. 如果我们使用 dup 复制 epoll 的文件描述符, 那么两个文件描述符指向同一个 epoll. 对两个文件描述符使用 epoll_wait 和 epoll_ctl 是等效的, 会造成同一 epoll 文件的修改.
  2. 这一情形同样发生在 fork 的时候, 子进程继承自父进程的 epoll 指向同一文件.
int epfd, fd1, fd2;
struct epoll_event ev;
struct epoll_event evlist[MAX_EVENTS];

/* Omitted: code to open 'fd1' and create epoll file descriptor 'epfd' ... */

ev.data.fd = fd1
ev.events = EPOLLIN;
if (epoll_ctl(epfd, EPOLL_CTL_ADD, fd1, ev) == -1)
	errExit("epoll_ctl");

/* Suppose that 'fd1' now happens to become ready for input */

fd2 = dup(fd1);
close(fd1);
ready = epoll_wait(epfd, evlist, MAX_EVENTS, -1);
if (ready == -1)
	errExit("epoll_wait");

所以上图的代码, 最后一个 ready 依然会返回 fd1 可读, 因为 fd2 还指向同一个文件描述, 所以 epoll 不会删除相应的信息. (应该算是一个 BUG ? 但是只要把它仔细地描述出来就是 FEATURE, 手动滑稽).

性能对比

可以看出 epoll 的性能十分优秀, 原因有主要有这两点:

  1. 对于 poll 和 select 调用, 内核每次都要检查所有注册的文件描述符,与之相对的 epoll 把感兴趣的事件注册到内核, 一旦事件发生就会通知内核中的 eventpoll 改变 readylist. epoll_wait 只需要把 wait_list 拉回来就行了.
  2. 对于 poll 和 select 调用, 每次都需要完整的传递感兴趣的文件描述符集合到内核, 而 epoll 则是使用 epoll_ctl 通知内核在内核空间建立数据结构. 大大减少了这一部分的时间成本.

因此 poll 和 select 对文件描述符的数量具有很强的敏感性, 所以执行时间近乎线性依赖 N 的大小. 与此对应的是, epoll 线性依赖与 I/O 时间的数量. 这一特性, 使得 epoll 在很多场景有着非常好的应用.

LT 和 ET

epoll 支持两种通知模式, level-triggered 和 edge-triggered. 这两个不知所云的概念其实借鉴自其他学科, 有一篇文章很好的解释了这两个概念:

https://zhuanlan.zhihu.com/p/20315482:

在epoll的man文档里，我们会看到一个花费大量篇幅描述的两个概念：

LT（Level Triggered，水平触发） 和 ET（Edge Triggered，边沿触发）

作者当年花费了九牛二虎之力也没能领悟这段“经文”。后来一个偶然的机会， 一个做电子设计的朋友给我讲明白了其中的道道。

为了弄明白LT（Level Triggered，水平触发） 和 ET（Edge Triggered，边沿触发）， 我们先要了解，这个Level和Edge是什么涵义，Level翻译成中文这里准确的涵义应该是电平； Edge是边沿。

这两个词曾经是电子信号领域的一个专有名词。如果，用时序图来标示一个数字电信号“010”， 应该是类似下图所示：

  • 低电平表示0。
  • 高电平表示1。
  • 0向1变化的竖线就是上升沿。
  • 1向0变化的竖线就是下降沿。
  • 在0或者1的情况下触发的信号就是LT（Level Triggered，水平触发）
  • 在0向1、1向0变化的过程中触发的信号就是 和 ET（Edge Triggered，边沿触发）

0或1都是一个状态，而0向1、1向0变化则只是一个事件。

我们很直观的就可以得出结论，LT是一个持续的状态，ET是个事件性的一次性状态。

二者的差异在于Level Triggered模式下只要某个socket处于readable/writable状态， 无论什么时候进行epoll_wait都会返回该socket；

而Edge Triggered模式下只有某个socket从unreadable变为readable或 从unwritable变为writable时，epoll_wait才会返回该socket。

虽然有很多资料表明ET模式的销量会比LT稍高， 但ET模式的编程由于事件只通知一次，很容易犯错误导致程序假死，我们推荐epoll工作于LT模式。 除非你很清楚你选择的是什么。

epoll 模式使用 LT 模式, 我们可以指定 EPOLLET 标志来启用 ET 模式:

struct epoll_event ev;
ev.data.fd = fd
ev.events = EPOLLIN | EPOLLET;
if (epoll_ctl(epfd, EPOLL_CTL_ADD, fd, ev) == -1)
	errExit("epoll_ctl");

举个例子展示一下 LT 和 ET 之间的差别, 假设我们使用 epoll 监视一个 socket 是否可以输入, 接下来发生的下面的情形:

  1. 输入抵达 socket
  2. 执行 epoll_wait. 这时不管是 LT 还是 ET 都会通知我们 socket 已经就绪.
  3. 再次执行 epoll_wait.

这是 LT 模式会告诉我们 socket 就绪, ET 会阻塞.

I/O模型总结

 Feb 2, 2019

概览

最直觉的 I/O 模型是, 一个进程在同一时刻对一个文件描述符进行 I/O 操作, 每个 I/O 系统调用阻塞直至数据传输完成. 但是这种简单的阻塞式的 I/O 模型在很多场景下并不适用.

  • 尽可能不阻塞地检查一个文件描述符是能进行 I/O.
  • 监视多个文件描述符.

于是乎, 就有了本文要讨论的这三种模型 :

  • I/O 多路复用 (select() 以及 poll() 系统调用)
  • 信号驱动 I/O
  • Linux 特有的 epoll() API

这三种方式实际上都是为了实现同一个目标, 同时检查多个文件描述符, 查看他们是否准备好了执行 I/O 操作. 这里的准备好, 实际上指的是在这个文件描述符上执行的 I/O 是非阻塞的(包括返回错误), 因为模型只是对文件描述符检查, 而不进行数据传输.

参考资料 : Manjaro 官方User Guide

  1. 官网下载镜像
  2. Linux下推荐通过命令行制作
    1. 插上用于制作安装盘的U盘
    2. 查看U盘盘符. sudo fdisk -l.
    3. 写入镜像: sudo dd if=manjaro-*.iso of=/dev/(替换成你查到的盘符) bs=4M
  3. Windows推荐使用Rufus:
    1. 选择你要用于制作安装盘的U盘.
    2. 在Boot selection, 点击Select选择下载的Manjaro镜像, 点击Start, 然后在出现的窗口中选择DD Image 方式.

函数与事件驱动处理

我们之前看到的系统设计都是为了长期运行的功能，服务器总是在线，这些模式对于那些处于高负载，占据大量内存和磁盘的服务来说是合适的。但是还有一类应用，可能需要在线的时间很短，或者只需要处理单一请求，或者只需要对每个时间响应。这种类型的请求或者叫事件驱动应用设计最近在大规模公有云上非常流行，厂家为此提出了FaaS（函数/功能即服务）。时间再往后推一点的最近，FaaS在私有云或者物理机器上也开始流行。

大多数时FaaS指无服务器（Serverless）计算，这么说也没什么错，但是广义的无服务计算和事件驱动FaaS并不是完全相同的概念。比如一个多用户容器（CaaS）是无服务器的，但是不是事件驱动的。一个开源的FaaS跑在你自己拥有管理的物理机上，这是事件驱动的但不是无服务器的。理解这两者的区别可以让你了解什么时候该使用事件驱动什么时候该使用无服务器，或者什么时候两者并用。

服务器模式

微服务

最近微服务变成了一个描述多节点分布式软件架构的流行词，微服务描述了一个系统由运行在不同进程上的组件组成，批次之间通过预定义的API进行交流。与微服务相对应的概念是宏系统，把所有的功能紧凑的放在一个应用中。

微服务有各种各样的好处，大多数情况下，微服务兼具可靠性和灵活性。微服务把一个应用分解成小片，每个小片专注于提供一种服务。这种模式有利于减小开发团队大小，使得开发更加灵活有方向性。
